# Git Pro

- Always inspect live state before advising or acting: `git status --short`, `git branch --show-current`, and relevant diffs/logs.
- Do not commit, push, reset hard, clean, rebase, or rewrite history unless the user explicitly asks.
- Preserve user changes. Distinguish staged, unstaged, untracked, ignored, and generated files.
- For reviews, cite `path:line` and use `git diff --stat`, targeted diffs, and history only as needed.
- For conflicts, understand both sides before resolving; avoid choosing ours/theirs blindly.
- Xcode `.xcodeproj` files are OpenStep property lists: when Xcode reports “not a valid property list” or “unresolved source control conflicts,” inspect `*.xcodeproj/project.pbxproj` directly. Search for conflict markers (`<<<<<<<`, `=======`, `>>>>>>>`) and malformed project syntax, but verify with `xcodebuild -list -project <Project>.xcodeproj` or `xcodebuild -project <Project>.xcodeproj -scheme <Scheme> -showBuildSettings`; generic `plutil -lint` may reject valid pbxproj header comments and is not the final authority. See `references/xcode-pbxproj-repair.md`.
- For bisect/debugging, keep steps reproducible and record good/bad commits and test command.
- When asked to "clean up history" or "make sure a branch only contains X-related changes" without further specifics, treat it as an AUDIT task, not a rewrite task: enumerate the full diff against the merge-base/target branch, classify every file/commit against the stated scope, and only rewrite history if you actually find out-of-scope commits. Do not assume a rewrite is needed just because the user used the word "clean up" — report "already clean" when that's the honest finding. See `references/audit-branch-scope.md` for the file-by-file causality technique (temporarily swapping a suspect file back to the base branch and rerunning the relevant linter/type-checker to prove whether a change is a genuine dependency of the feature or unrelated scope creep) and for cleaning up your own temporary edits afterward.
- Verify repository health after changes with status and the relevant tests/builds.
- After build/test verification, inspect whether generated artifacts changed. If build outputs are tracked or unignored (for example `.build/` in SwiftPM repos), report that noise separately and do not run cleanup/reset commands unless the user explicitly approves that destructive cleanup scope.
- For dotfile/configuration repositories, classify dirty files before cleanup: tracked user configuration changes, generated tool state, editor noise, and machine-specific/secret-adjacent values. Prefer adding narrowly-scoped ignore rules for reproducible generated state (for example lock/usage metadata) and removing untracked generated files; do not reset tracked dotfile edits unless the user explicitly asks.
- When removing generated files such as `.DS_Store` from git, check both tracking and ignore state: use `git ls-files '*DS_Store*'` to find tracked metadata, remove tracked copies with `git rm`/`git rm --cached` as appropriate, ensure an ignore rule covers future copies, and verify with `git check-ignore -v` plus a final `git ls-files` check.
- When validating a dirty config cleanup, run the narrow validators for touched config types (for example shell syntax, editor settings parse checks, package-manager dry checks) and report any unmet dependencies separately from the repository cleanup.
- When asked to create a clean commit history from dirty config files, validate first, then stage and commit by logical concern rather than by incidental file order. If the branch is both ahead and behind, fetch/rebase before pushing, confirm the local branch is only ahead of upstream, push, and finish with commit subjects plus clean/aligned status. See `references/dotfile-clean-commit-history.md` for a concrete checklist.
- When committing a requested subset while unrelated local changes exist, stage explicit pathspecs only and leave unrelated files unstaged. If push preparation needs remote state, prefer `git fetch` plus ahead/behind inspection over `git pull --rebase`, because rebase/pull refuses a dirty worktree and may tempt unsafe stashing or accidental inclusion. Do not stage generated memory/profile files unless the user explicitly asks to commit them.
- When asked to fold a fix into an *earlier* commit (e.g. "the seed script commit should already do X") rather than append a new fixup commit on top, use `git rebase -i <base>` with that commit set to `edit`, apply the change, amend, then `git rebase --continue`. This keeps history scoped to logical units instead of "add feature" + "fix feature" pairs. See `references/amend-earlier-commit.md` for the exact procedure and the pre-commit-hook reformatting pitfall.
- Pre-commit hooks operate on the union of staged files and tracked-but-dirty files in the worktree. During commit-surgery, that can rewrite a dirty file that you deliberately left unstaged — producing a `MM` (matched-modified in index, modified in worktree) boundary. Before retrying, inspect `git status --short`, re-stage the rewritten file if it still belongs in that commit, and unstage any files that were accidentally included. See `references/commit-surgery-checklist.md` for the exact sequence.
- After `git reset --soft <base>` for commit surgery, all local changes remain staged. Before committing, always verify the staged pathspec with `git diff --cached --name-only`, then unstage any files that belong in a different commit with `git restore --staged -- <path>`. Telling git to commit "the staged tooling changes" and then discovering the editor files were also staged is the root cause of accidental boundary violations. See `references/commit-surgery-checklist.md` for a safer sequencing.
- Never push a rewritten/rebased branch without the user's explicit go-ahead, even if they asked for the rewrite itself — rewriting local history and force-pushing over a shared/tracked branch are separate approvals. Tell the user their branch has diverged from `origin/<branch>` and that a `git push --force-with-lease` is needed, but let them decide when.
