# Python API Pro

Use this skill for Python services built with FastAPI, SQLAlchemy, Alembic, and Pydantic.

## External docs fallback

If this skill does not cover a requested FastAPI, SQLAlchemy, Alembic, Pydantic, or Python API best practice, search the relevant official docs before answering or editing code. Match the project's pinned versions from its dependency files, and cite any external guidance used.

- [FastAPI documentation](https://fastapi.tiangolo.com/)
- [SQLAlchemy documentation](https://docs.sqlalchemy.org/)
- [Alembic documentation](https://alembic.sqlalchemy.org/)
- [Pydantic documentation](https://docs.pydantic.dev/)
- [Python documentation](https://docs.python.org/3/)

## Default workflow

1. Inspect the project first: `pyproject.toml`, dependency manager, app entrypoint, DB/session setup, model/schema layout, tests, and existing style.
2. Trace symbols to definitions and usages before editing: routers, dependencies, SQLAlchemy models, Pydantic schemas, migrations, and tests.
3. Prefer small changes with tests. Run the narrowest relevant test first, then broader tests if the change touches shared API/database infrastructure.
4. Do not invent packages. Use dependencies already present unless the user approves adding one.

## FastAPI conventions

- Keep routers thin: request parsing, dependency injection, auth checks, and response mapping belong there; business logic belongs in services/use-cases.
- Use dependency functions for database sessions, current user/auth, settings, and request-scoped clients.
- Prefer explicit `response_model` or return annotations; avoid leaking ORM objects unless the project already uses `from_attributes=True` schemas consistently.
- For async apps, do not call blocking DB/network/file code in the event loop. For sync SQLAlchemy, keep endpoints sync unless the app’s existing pattern says otherwise.
- Raise `HTTPException` at the API boundary; keep domain/service functions framework-agnostic where practical.
- Add OpenAPI-friendly types and examples only when they improve consumer clarity.

## SQLAlchemy 2.x conventions

- Prefer 2.0-style `select()`, `Session.execute()`, `scalars()`, `scalar_one_or_none()`, and typed `Mapped[...]` / `mapped_column()` when the codebase uses modern SQLAlchemy.
- Keep transaction boundaries explicit. Use `session.begin()` or a unit-of-work pattern; avoid hidden commits in low-level helpers unless it is an established project convention.
- Avoid N+1 queries: inspect relationship loading and use `selectinload`, `joinedload`, or explicit joins intentionally.
- For async SQLAlchemy, use `AsyncSession`, `await session.execute(...)`, and avoid lazy-loading patterns that perform implicit IO.
- Validate cascade, nullable, uniqueness, and index behavior against the database schema and migrations, not only ORM declarations.

## Alembic conventions

- Every model/schema change that affects persistence needs a migration unless the project explicitly does schema management another way.
- Review autogenerated migrations manually: names, nullable transitions, default/backfill strategy, indexes, constraints, enum changes, and downgrade behavior.
- For existing data, make migrations safe in phases: add nullable/default column, backfill, enforce non-null/constraints in a later step when necessary.
- Never drop/rename columns or tables without confirming data migration and rollback expectations.
- Test migrations with `alembic upgrade head` and, where feasible, downgrade/upgrade on a disposable database.

## Pydantic v2 conventions

- Use `BaseModel`, `ConfigDict`, `field_validator`, `model_validator`, `computed_field`, and `model_dump()`/`model_validate()` idioms for v2.
- Use `from_attributes=True` for ORM-to-schema conversion when returning ORM-backed objects.
- Separate create/update/read schemas when validation, optional fields, or exposed fields differ.
- Keep secrets as `SecretStr`/`SecretBytes` where appropriate and avoid printing them in logs/tests.
- Prefer explicit timezone-aware datetime handling and clear decimal/money representations.

## Review checklist

- API behavior: status codes, error shapes, authz/authn, pagination/filtering/sorting, idempotency.
- Data behavior: transaction scope, uniqueness/races, eager loading, migrations, backfills, rollback story.
- Validation: Pydantic v2 idioms, coercion vs strict validation, public/private fields.
- Observability: structured logging without secrets, useful error context, metrics/tracing if present.
- Tests: unit tests for services, API tests for endpoints, DB tests for migrations/queries where relevant.
